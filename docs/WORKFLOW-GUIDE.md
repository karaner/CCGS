## 阶段 0: 设置和配置

### 你需要什么

开始之前确保你有：

- **Claude Code** 已安装并正常工作
- **Git** 配合 Git Bash（Windows）或标准终端（Mac/Linux）
- **jq**（可选但推荐——如果缺少，钩子会回退到 `grep`）
- **Python 3**（可选——某些钩子使用它进行 JSON 验证）

### 步骤 0.1: 克隆和配置

克隆仓库并在编辑器中打开：

```bash
git clone <repo-url> my-game
cd my-game
```

### 步骤 0.2: 运行 /start（新用户推荐）

如果你对项目不熟悉或还不知道要做什么游戏：

```
/start
```

这个引导式入职会问你现在在哪里（没有想法、模糊想法、清晰想法、现有工作）并引导你到正确的阶段。如果已经有游戏概念和引擎决策，跳过此步骤。

### 步骤 0.3: 选择你的引擎

在 Claude Code 中运行 `/setup-engine`。这是最重要的配置步骤——它告诉每个 Agent 你使用什么引擎、语言和工具链：

```bash
/setup-engine unity 2023.2
```

**`/setup-engine` 做什么：**

- 在 `CLAUDE.md` 中固定引擎和版本
- 填充 `.claude/docs/technical-preferences.md`，包括命名规范、性能预算和引擎特定默认值
- 检测知识差距（引擎版本是否比 LLM 训练数据新）并从网络获取当前文档，以便 Agent 提供正确的 API 建议
- 在 `docs/engine-reference/` 中创建版本固定参考文档

### 步骤 0.4: 验证钩子是否正常工作

启动一个新的 Claude Code 会话。你应该看到 `session-start.sh` 钩子的输出：

```
=== Claude Code 游戏工作室 -- 会话上下文 ===
分支: main
最近提交:
  abc1234 初始提交
===================================
```

如果看到此信息，说明钩子正常工作。如果没有，检查 `.claude/settings.json` 确保钩子路径对你的操作系统是正确的。

## 阶段 1: 构思和概念

### 此阶段发生什么

你从"没有想法"或"模糊想法"变成结构化的游戏概念文档。这是你弄清楚**做什么**的阶段。

### 步骤 1.1: 使用 /brainstorm 进行头脑风暴

这是你跳过 `/start` 后的起点。运行 brainstorm 技能：

```
/brainstorm
```

或带类型提示：

```
/brainstorm roguelike deckbuilder
```

**发生什么：** brainstorm 技能使用专业工作室技术引导你完成协作的 6 阶段构思过程。

### 步骤 1.2: 记录获胜概念

采用 brainstorm 输出并将其正式化。使用**游戏概念模板**。

### 步骤 1.3: 定义你的游戏支柱

在进一步之前，锁定你的**游戏支柱**——指导每个决策的 3-5 个不可协商的设计价值。

> 当你对概念和支柱满意时，你就准备好进入预生产了。

## 阶段 2: 预生产和设计

### 此阶段发生什么

你创建定义游戏如何运作的所有设计文档。在此阶段什么都不编码——这是纯粹的设计和架构。

### 步骤 2.1: 将概念分解为系统

在编写单独的 GDD 之前，列举游戏需要的所有系统：

```
/map-systems
```

这会创建 `design/gdd/systems-index.md`——一个主跟踪文档，列出：
- 游戏需要的每个系统
- 系统之间的依赖关系
- 分配优先级层（MVP、垂直切片、Alpha、完整愿景）
- 确定设计顺序

然后按依赖顺序设计每个系统：

```
/map-systems next
```

这会选择最高优先级未设计系统并交接给 `/design-system`。

### 步骤 2.2: 创建游戏设计文档（GDD）

为游戏的每个主要系统在 `design/gdd/` 中创建设计文档。

**`design-docs` 规则**要求每个 GDD 有以下 8 个部分：

| # | 部分 | 内容 |
|---|------|------|
| 1 | **概述** | 系统的一段落总结 |
| 2 | **玩家幻想** | 玩家使用此系统时的想象/感受 |
| 3 | **详细规则** | 明确的机械规则——如果两个设计师阅读，他们会构建相同的东西 |
| 4 | **公式** | 每个计算，包含变量定义和范围 |
| 5 | **边缘情况** | 异常情况会发生什么？明确解决。 |
| 6 | **依赖** | 连接到的其他系统（双向） |
| 7 | **调节旋钮** | 设计师可以安全更改的数值，包含安全范围 |
| 8 | **验收标准** | 如何测试这是否有效？具体、可衡量。 |

### 步骤 2.3: 设计审查

在任何设计文档交给程序员之前，运行审查：

```
/design-review design/gdd/crafting-system.md
```

## 阶段 3: 原型和验证

### 此阶段发生什么

在承诺全面生产之前，你用可丢弃的原型验证有风险或不确定的机制。这可以节省数周的浪费生产工作。

### 步骤 3.1: 识别需要原型的东西

不是所有东西都需要原型。当以下情况时进行原型设计：
- 一个机制是新颖的，不确定它是否有趣
- 一个技术方法有风险，不确定是否可行
- 两个设计选项看起来都可行，需要感受差异

### 步骤 3.2: 运行原型技能

```
/prototype "grappling hook movement with momentum"
```

### 步骤 3.3: 评估原型

玩/测试原型并回答：
- 假设成立吗？有趣/可行/更好吗？
- 什么让你惊讶？
- 生产中会改变什么？

更新原型 README 并给出**裁决**：发货、返工或取消。

> 当你的原型验证了（或推翻）了你最冒险的假设时，你就准备好进入生产了。

## 阶段 4: 生产冲刺工作流

### 此阶段发生什么

这是核心生产循环。你按冲刺（通常 1-2 周）工作，实现功能、修复错误和跟踪进度。此阶段重复直到你的游戏内容完整。

### 步骤 4.1: 创建你的第一个冲刺

```
/sprint-plan new
```

### 步骤 4.2: 每日工作循环

每个会话遵循此模式：

```
1. 启动 Claude Code 会话
2. 从冲刺计划中选择一个任务
3. Claude 识别正确的 Agent 来帮助
4. 实现功能
5. 编写测试（验证驱动开发）
6. 提交你的工作
7. 重复直到会话上下文变满
```

### 步骤 4.3: 冲刺跟踪

随时检查冲刺进度：

```
/sprint-plan status
```

### 步骤 4.4: 冲刺审查和回顾

在冲刺结束时：

```
/retrospective
```

然后计划下一个冲刺：

```
/sprint-plan new
```

## 阶段 5: 实现深度

### 此阶段发生什么

这是代码被编写的地方。Agent 系统通过路径特定规则、自动化钩子和结构化委托确保代码质量。

### 步骤 5.1: 单系统实现

对于单个游戏系统：

```
让 gameplay-programmer agent 根据 design/gdd/combat-system.md 中的规定实现伤害计算系统。
```

### 步骤 5.2: 多系统功能（团队协调）

对于跨多个领域的功能，使用团队技能：

```
/team-combat "healing ability with HoT (heal-over-time) and cleanse"
```

### 步骤 5.3: AI 系统

对于 NPC 行为、寻路或决策：

```
让 ai-programmer agent 实现敌人 NPC 的巡逻/调查/追逐行为树。
```

### 步骤 5.4: 引擎特定工作

当你需要引擎特定专业知识时：

```
# 对于 Unity：
让 unity-specialist agent 审查我们的场景树架构并提出性能和可维护性改进建议。
```

## 阶段 6: 测试和质量保证

### 此阶段发生什么

测试贯穿生产，而不仅仅在最后。系统支持测试驱动开发和自动化执行。

### 步骤 6.1: 先写测试（验证驱动开发）

项目强制**验证驱动开发**：在实现之前或同时编写测试，而不是之后。

### 步骤 6.2: 错误报告

当你发现错误时：

```
/bug-report "治疗法术在目标有护盾 buff 时反而造成伤害"
```

### 步骤 6.3: 测试规划

对于里程碑验证：

```
让 qa-lead agent 为 Alpha 里程碑创建测试计划，覆盖战斗、进度和 UI 系统。
```

### 步骤 6.4: 平衡测试

在实现经济或进度系统后：

```
/balance-check assets/data/combat_damage.json
```

## 快速开始工作流

### 工作流 1: "我刚起步，没有游戏想法"

```
1. /start
2. 从 brainstorm 输出中选择最佳概念
3. 创建游戏概念文档
4. 定义游戏支柱
5. /design-review 你的概念文档
6. /map-systems 将概念分解为带依赖和优先级的系统
7. /design-system 创作每个系统的 GDD
```

### 工作流 2: "我有一个设计，想开始编码"

```
1. /design-review 每个 GDD 确保它们是扎实的
2. 为你的第一个主要技术选择创建 ADR
3. /sprint-plan new 计划你的第一个冲刺
4. 开始实现
5. 每个主要功能后 /code-review
6. 随代码编写测试
7. 频繁提交
```

### 工作流 3: "我需要添加一个复杂功能"

```
1. 在 design/gdd/ 中创建/更新该功能的设计文档
2. /design-review 验证设计
3. /estimate 了解工作量和风险
4. 使用适当的 /team-* 技能
5. /code-review 实现
6. 如果影响游戏平衡，进行 /balance-check
```

## 从系统获得最多的提示

1. **总是从设计开始，然后实现。** Agent 系统构建在假设设计文档在代码编写之前存在。
2. **对跨领域功能使用团队技能。** 不要试图自己手动协调 4 个 Agent。
3. **信任规则系统。** 当规则在你的代码中标记某些东西时，修复它。
4. **主动压缩。** 在约 65-70% 上下文使用时压缩或 `/clear`。
5. **使用正确层级的 Agent。** 不要让 `creative-director` 写着色器。
6. **在将设计交给程序员之前运行 `/design-review`。** 这可以及早发现不完整的规格。
7. **每个主要功能后运行 `/code-review`。** 在架构问题蔓延之前抓住它们。
8. **先原型化有风险的机制。** 一天的原型可以节省一周的生产时间，如果机制不起作用的话。
9. **保持你的冲刺计划诚实。** 定期使用 `/scope-check`。范围蔓延是独立游戏的第一杀手。
10. **用 ADR 记录决策。** 未来的你会感谢现在的你记录了*为什么*事情是这样构建的。
